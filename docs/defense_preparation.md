# 答辩准备指南 (Defense Preparation Guide)

## 一、 答辩演讲逻辑框架（建议时长：10-15分钟）

**1. 项目背景与痛点痛点 (What & Why)**
*   **业务需求**：在电商、社交媒体、产品反馈中，企业每天都会产生海量中文评论。人工筛选这些数据耗时耗力，且主观性强、难以量化。
*   **解决方案**：我们开发了“云析智研”平台，这不仅是一个跑模型的脚本，而是一套**完整的 SaaS 级业务闭环系统**。它将深度学习大模型的能力“封装”成易用的界面，提供自动分析、图表可视化、人工纠偏和模型再训练的完整数据链路。

**2. 核心功能展示 (Features)**
*   **多维数据录入**：支持单条文本测试和 Excel/CSV 大批量文件导入。
*   **工作流流转与权限控制 (RBAC)**：
    *   *普通用户*：提交分析，查看基础结果。
    *   *分析师*：对机器分析结果进行人工审核纠偏（Human-in-the-loop），确保最终报告的准确性。
    *   *管理员*：管理底层大模型、调度数据集、发起模型微调（训练）任务。
*   **数据可视化与导出**：通过 ECharts 进行多维度看板展示，支持 ReportLab 生成 PDF 和 Excel 报表导出。

**3. 系统架构与技术选型 (Architecture & Tech Stack)**
*   **整体架构**：前后端分离架构 + 算法异步调度架构。
*   **前端**：Vue 3 + Vite + Element Plus + Tailwind CSS。采用现代单页应用方案，状态由 Pinia 管理。
*   **后端**：Django 6 + DRF。利用 Django 强大的 ORM 和生态快速构建企业级后台。
*   **异步调度**：引入 Celery + Redis。**（亮点：这是防阻塞的关键设计，向评委强调）**
*   **AI 算法层**：PyTorch + Transformers，支持本地加载预训练模型并进行前向推理和后向微调。

**4. 核心技术难点与解决方案 (Challenges & Solutions)**
*   **难点一：AI 推理耗时长，容易导致 Web 请求超时**。
    *   *解决*：引入了 Celery 分布式任务队列，将批量分析和模型训练任务放入后台异步执行，前端通过轮询或状态标记获取进度，不阻塞主线程。
*   **难点二：模型与 Web 服务的强耦合导致系统臃肿**。
    *   *解决*：在代码层面将 `sentiment_server/apps`（Web业务）和 `ml_assets/workspace`（算法工作区）物理隔离。Web 端只负责数据持久化和分发任务，算法脚本纯粹负责计算。

**5. 总结与未来展望 (Future Work)**
*   总结项目完成度高，是一套企业级的闭环系统。
*   未来可扩展：支持更多分类维度（如方面级情感分析 Aspect-Based Sentiment Analysis）、引入大语言模型（LLM/DeepSeek/Qwen）做更深度的文本总结，或引入 WebSocket 实时推送分析进度。

## 二、 评委高频 Q&A 预测及回答策略

**Q1：为什么后端选择 Django 而不是轻量级的 Flask 或运行更快的 FastAPI？**
**回答策略**：本项目不仅仅是一个简单的 API 接口，它涉及复杂的 RBAC 权限（用户/分析师/管理员）、操作审计、以及高度关系型的数据表（用户、任务、报告、审核记录）。Django 自带了完善的 ORM、Auth 系统以及强大的后台管理生态，能极大地提升此类“重业务逻辑”的企业级 SaaS 平台的开发效率。

**Q2：当分析任务非常大的时候（比如用户上传了包含十万条评论的 Excel），系统怎么保证不崩溃？**
**回答策略**：系统采用了**异步任务架构**。用户上传文件后，Web 容器（Django）只会将文件存储，并向 Celery 消息队列（基于 Redis）投递一个任务 ID 后立即返回给前端。由后台的 Celery Worker 监听队列并慢慢处理这十万条数据，处理完成后更新数据库状态。前端只需根据任务 ID 获取状态即可。

**Q3：项目里的“人工纠偏/审核”功能有什么实际意义？**
**回答策略**：这是工业落地的一个重要考量（Human-in-the-loop）。任何模型都无法做到 100% 准确，特别是在特定领域的黑话或反讽语境中。分析师对判断错误的样本进行人工纠偏后，这些数据会作为新的“高质量标注数据”存入系统数据集。管理员可以基于这些修正后的数据，在后台直接发起 Celery 训练任务对模型进行微调（Fine-tuning），从而形成“越用越准”的数据飞轮闭环。

**Q4：前端为什么同时使用了 Element Plus 和 Tailwind CSS？**
**回答策略**：Element Plus 提供了重度交互的复杂组件（如数据表格、分页、模态框等），可以快速搭建后台功能；而 Tailwind CSS 提供的是原子化的工具类。用 Tailwind 可以极速实现自定义布局、精细的间距调整和定制化的深色侧边栏 SaaS 风格，两者结合既保证了开发效率，又保证了界面的高度定制化和美观。

**Q5：系统安全性是如何考虑的？**
**回答策略**：
1. **接口安全**：采用 DRF SimpleJWT，前后端通过 Token 鉴权，无状态且防伪造。
2. **数据安全**：前后端分离，前端使用 DOMPurify 防御 XSS 攻击。
3. **权限隔离**：后端在视图层严格判断角色，普通用户绝对无法调用训练或审核接口。
4. **日志审计**：系统的关键操作都有操作审计日志，可追溯。
